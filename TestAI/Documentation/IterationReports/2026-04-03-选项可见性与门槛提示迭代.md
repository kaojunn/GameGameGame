# 迭代报告：2026-04-03 — 选项可见性与门槛提示

## 1. 迭代前玩法（当时是什么样）

- NorthernTown2026 是运行时构建 UGUI 的文字冒险：左侧叙事日志、右侧状态/装备/背包、底部选项按钮。
- 玩家通过点击选项推进节点；部分选项有检定（属性 + 1d10），部分有门槛（需要物品、需要洞察+机巧总和）。
- **问题点**：门槛不满足时，选项会被直接隐藏，玩家不知道“还有哪些路线”和“差了什么条件”，分支可读性偏弱。

## 2. 趋势要点（外部检索浓缩）

1. **互动叙事强调“可意图选择（intentional choices）”**  
   Choice of Games 的设计文章指出：在高风险或有难度差异的选项前，应该让玩家明确潜在结果、相关检定与相对难度，避免“误解意图后挫败”。  
   来源语境：How to Write Intentional Choices（Choice of Games，2016）  
   https://www.choiceofgames.com/2016/12/how-to-write-intentional-choices/

2. **叙事型反复游玩要减少“被难度墙卡剧情”的体验**  
   GDC Podcast 对 Hades 创作经验的访谈提到：系统设计应尽量避免玩家因为机制门槛而失去叙事推进动机。  
   来源语境：GameDeveloper 访谈（2021）  
   https://gamedeveloper.com/design/roguelikes-and-narrative-design-with-i-hades-i-creative-director-greg-kasavin

3. **分支选择 UX 常见做法：保留选项可见性并解释不可达原因**  
   近年的 choice-based 设计讨论普遍强调，透明的条件反馈（而非静默隐藏）更有利于玩家规划后续路线与重玩目标。  
   来源语境：choice-based UX 相关文章集合检索（用于方向归纳）。

## 5. 验收自检（Play 模式/逻辑推演）

- 当选项缺少 `RequiresItemId` 或 `RequiresInsightSum` 条件时：按钮依然显示，但不可点击，且有明确原因文本。
- 带 `StatCheck` 的可选项：显示检定提示 `Stat + 1d10 ≥ Threshold`，并展示当前属性值。
- 可选项点击流程、经验结算、检定跳转、拖拽装备/消耗流程未改动，回归风险可控。
- 本次未新增 PlayerPrefs 键，不涉及持久化键冲突。

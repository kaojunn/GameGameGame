# 迭代报告：2026-04-03 — NorthernTown2026 快速重开体验

## 1. 迭代前玩法（当时是什么样）

基于仓库现有代码与场景（`MainTextAdventure.unity` + `TextAdventureBootstrap` 运行时建 UI）还原如下：

- **玩家操作**
  - 点击底部选项按钮推进节点（`TextAdventureEngine.Choose`）。
  - 右侧背包卡片可拖放到装备槽 / 使用区 / 背包区（装备、消耗、卸下）。
- **目标**
  - 通过节点选择与检定推进分支，达成不同结局（如 `ending_public`、`ending_leave` 等）。
- **失败 / 胜利定义**
  - 无传统 HP 失败；“失败”主要体现在检定失败进入不同分支。
  - 到达结局节点即一轮叙事闭环完成，可“再玩一次”回到 `start`。
- **进度 / Meta**
  - 已有本局计数：`RunNodesVisitedCount`、`RunChoicesCount`（在状态区展示）。
  - 有等级、经验、装备属性加成；无局外长期资源。
- **反馈（UI/提示）**
  - 左侧日志滚动展示叙事与系统信息（检定细节、经验、装备提示）。
  - 右侧状态区展示属性、装备、背包。
  - **缺口**：除“结局后点再玩一次”外，缺少“任意时刻重开本局”的低摩擦入口。

## 2. 趋势要点（外部检索浓缩，2~3 个方向）

1. **短局/多周目玩法强调低摩擦重开**
   - 语境：Slay the Spire 2 社区模组 “Quick Restart” 用长按 R 快速重开当前 run（保留角色/难度等框架，重置局内过程）。
   - 参考：`https://sts2mods.com/quick-restart/`

2. **概率/检定类玩法常见“信息显式化”**
   - 语境：BG3 骰子检定体系强调 DC、掷骰与修正可见，帮助玩家理解决策与结果。
   - 参考：`https://bg3.wiki/wiki/DC`

3. **策略游戏社区对“结果信息可见”有持续需求**
   - 语境：XCOM 社区出现“显示命中率”类模组，反映玩家对可解释反馈和可复盘性的偏好。
   - 参考：`https://steamcommunity.com/sharedfiles/filedetails/?id=621959270`

## 3. 对比结论（为什么只选这一项）

| 方向 | 与当前项目差距 | 适不适合这次小迭代 |
|------|----------------|--------------------|
| A. 增加“任意时刻快速重开”入口 | 当前仅靠到达结局后“再玩一次”，中途重试路径长 | **适合，选中**（改动集中在引擎与 UI 入口，验收标准明确） |
| B. 强化检定可视化（例如选项上直接显示阈值标签） | 需改选项文案生成与布局，涉及 UI 密度调优 | 暂不选（本轮优先更小改动面） |
| C. 新增更复杂的 Meta（局外解锁/持久化） | 需设计持久化结构和回归验证，侵入面较大 | 暂不选 |

**本次只实现 A：快速重开本局（按钮 + R 键）**。

## 5. 验收自检（Play 模式或逻辑推演）

- **预期行为**
  - 任意节点按 `R` 或点“重新开始本局 (R)”按钮：
    - 触发 `RestartRun`；
    - 清空日志（`OnNewRunStarted -> ClearLog`）；
    - 玩家状态重置（等级/经验/装备/背包回到开局）；
    - 重新进入 `start`，并继续正常生成可选项。
- **回归关注**
  - 拖拽中的装备卡在 UI 外：重开后会走统一 `RefreshAll` 重建卡片，避免状态残留。
  - 原有“结局后再玩一次”路径：现在同样走 `RestartRun`，逻辑更一致。
- **持久化冲突**
  - 本次未新增 `PlayerPrefs` 或外部存档键名，不存在键名冲突风险。

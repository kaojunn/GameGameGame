# 迭代报告：2026-04-03 — 一键重新开局（NorthernTown2026）

## 1. 迭代前玩法（当时是什么样）

- 当前为运行时构建 UI 的文字冒险：左侧叙事日志、底部选项区、右侧状态/装备/背包/使用区。
- 玩家通过点击选项推进节点，进行属性检定（属性 + 1d10）并获取经验、物品、装备加成。
- 已有「周目重置」能力，但触发方式主要是剧情中某些结局节点的「再玩一次」选项跳回 `start`。
- 缺口：当玩家在中途想直接重开，不够直观；需要先走到特定节点或依赖剧情路径，复盘效率不高。

## 2. 趋势要点（外部检索浓缩）

1. **互动叙事常见 QoL：显式重开入口**  
   在 Twine/SugarCube 生态中，`UI.restart()` / `Engine.restart()` 是常见做法，强调“重开”作为独立操作而非仅回到首节点。  
   语境来源：Twine Q&A（SugarCube restart 讨论）。

2. **状态连续性与“真正重置”要区分**  
   Chapbook 文档强调，普通跳回首段不等于重置变量；应使用专门的 restart 机制避免残留状态。  
   语境来源：Chapbook 文档《Continuity Between Sessions》。

3. **分支叙事也在强化 checkpoint / restart 体验**  
   ChoiceScript 社区已有 checkpoints 功能讨论与发布语境，说明“重玩/重开便利性”是持续需求。  
   语境来源：Choice of Games Forum（checkpoints 公告与讨论）。

## 3. 对比结论（为什么选这一项）

| 方向 | 与当前项目差距 | 本次是否适合 |
|---|---|---|
| A. 一键重新开局按钮 | 仅需引擎公开重开接口 + 右栏增加 1 个按钮 | **选中**（改动面小、依赖少、验收清晰） |
| B. 检查点/存档系统 | 需设计状态快照、数据结构与 UI，改动较大 | 否 |
| C. 成就/局外 Meta | 需新增追踪字段、持久化与展示逻辑 | 否 |

本次只实现 A，目标是让玩家在任意节点都能快速进入新局，且保证状态彻底重置。

## 6. 验收自检（逻辑推演）

- 任意剧情节点点击“重新开局”后：日志清空、玩家状态重置、重新显示 `start` 节点文本。
- 本局计数（已读节点/已做选择）在重开后从新局重新累计，不延续旧局。
- 装备/背包回到 `ResetForNewGame` 定义的开局状态（含开局面包），无残留装备状态。

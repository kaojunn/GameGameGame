# 迭代报告：2026-04-03 — 检定成功率可视化

## 1. 迭代前玩法（当时是什么样）

- **操作**：点击选项推进剧情；拖拽背包卡片到装备槽/使用区完成装备、卸下、消耗。
- **目标**：在分支叙事中做选择，达成不同结局（公开曝光、离开小镇、隐藏结局等）。
- **失败/胜利**：没有传统战斗失败界面；通过剧情分支进入不同结局节点并可再开一局。
- **进度/Meta**：有等级、经验、四维属性、装备与背包；本局已读节点/已做选择会显示在状态面板。
- **反馈**：左侧叙事日志（含检定掷骰结果）、底部选项按钮、右侧状态/装备/背包 UI。  
  **缺口**：玩家在点检定选项前，看不到当前成功概率预期，只能点下去后才知道结果。

## 2. 趋势要点（外部检索浓缩）

1. **“把影响结果的关键信息前置显示”**  
   语境：开发者 HUD 复盘文章强调，不展示关键数值会削弱玩家对系统的理解与信任。  
   来源：DEV Community — *Your Dungeon Crawler HUD Is Lying By Omission*  
   https://dev.to/meridian-ai/your-dungeon-crawler-hud-is-lying-by-omission-1f35

2. **“检定界面应突出难度与角色加值”**  
   语境：围绕 BG3 检定 UI 变更的讨论，强调 DC 与加值可视化能增强‘我为何成功/失败’的可解释性。  
   来源：Gaming Respawn — *Baldur’s Gate III’s New Dice-Rolling Interface...*  
   https://gamingrespawn.com/features/54148/baldurs-gate-iiis-new-dice-rolling-interface-truly-captures-the-feeling-of-dd-ability-checks/

3. **“技能检定要有可感知的策略空间与信息表达”**  
   语境：Unexplored 2 设计复盘中，反复强调检定系统的可理解性与玩家决策参与感。  
   来源：GameDeveloper — *Designing Unexplored 2's Fortune System*  
   https://www.gamedeveloper.com/design/a-table-top-feel-designing-unexplored-2-s-fortune-system-to-replace-virtual-dice-rolling

## 3. 对比结论（为什么选这一项）

| 方向 | 与当前项目差距 | 这次是否适合小迭代 |
|---|---|---|
| A. 检定选项显示成功率 | 仅缺一层 UI 文案计算；底层检定规则已存在 | **是（采纳）** |
| B. 结局统计持久化（PlayerPrefs） | 需新增存档键、重置策略、展示位 | 否（改动面更大） |
| C. 失败后快速重试按钮 | 需改流程语义与节点控制，可能影响叙事节奏 | 否（验收边界不够小） |

本次只选 **A：检定选项显示成功率**。原因：改动集中在按钮文本生成，依赖少，验收标准清晰。

## 4. 本次改动列表（改了哪些文件/行为）

### 代码改动

- `Assets/NorthernTown2026/Scripts/TextAdventureBootstrap.cs`
  - 在选项按钮文案生成处引入 `FormatChoiceLabel(ChoiceOption option)`。
  - 对含 `StatCheck` 的选项追加文本：`（成功率 xx%）`。
  - 新增 `ComputeCheckSuccessChancePercent(int stat, int threshold)`，按当前规则 `属性 + 1d10 >= 阈值` 计算概率。

### 行为变化

- 检定选项在显示阶段即给出当前角色属性下的成功率预估（0%~100%，10% 步进）。
- 非检定选项文案保持不变。

## 6. 验收自检

- 逻辑推演：
  - 若 `stat=3, threshold=12`，需掷出 `9/10`，成功率应为 `20%`；
  - 若 `stat=3, threshold=11`，需掷出 `8/9/10`，成功率应为 `30%`；
  - 文案应与上述一致，且不会影响实际掷骰流程。
- 回归关注：
  - 仅改选项文本，不改 `TextAdventureEngine.Choose()` 的结算逻辑；
  - 装备拖拽、消耗品使用、状态栏刷新路径不受影响。

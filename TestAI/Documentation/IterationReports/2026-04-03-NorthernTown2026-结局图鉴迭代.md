# 迭代报告：2026-04-03 — NorthernTown2026 结局图鉴小迭代

## 1. 迭代前玩法（当时是什么样）

- 玩法核心是文字分支：节点文本 + 选项推进，带属性检定（属性 + 1d10）。
- 角色有等级、经验、装备三槽（终端/外套/饰品）、背包与消耗品使用区（拖放交互）。
- 已有本局进度（已读节点/已做选择）展示，但**跨周目的 Meta 追踪较弱**：玩家达成过哪些结局，下一轮很难在界面上直接复盘。
- 胜负形态主要由不同结局节点表达（如 `ending_public`、`ending_hidden` 等），不存在传统 HP 归零失败判定。

## 2. 趋势要点（外部检索浓缩）

1. **分支叙事常见结构强调“重玩可见性”**  
   IFWiki 对 CYOA/选择叙事结构的总结提到，诸如 Time Cave、Loop and Grow 等模式天然鼓励重玩；当内容广度高时，玩家往往不会在单次流程看到全部分支，因此“已探索内容可见化”很关键。  
   来源语境：IFWiki《Standard Patterns in Choice-Based Games》。

2. **社区实践里“跨周目结局追踪”是常见需求**  
   Twine 开发者讨论明确提出“标题页显示已达成结局”的需求，并给出通过成就/持久化机制保留进度的方向，说明这是互动叙事里非常实用且低成本的 Meta 能力。  
   来源语境：Interactive Fiction Community Forum 贴文《Tracking which endings have been achieved across playthroughs》。

3. **平台标签层面，玩家对“选择有后果/多结局”有明确预期**  
   Steam 长期存在 “Choices Matter” 与 “Multiple Endings” 标签页，反映这类特征已成为玩家检索和预期的一部分。  
   来源语境：Steam Tags 页面（Choices Matter / Multiple Endings）。

## 3. 对比结论（为什么只选这一项）

| 范式/趋势 | 与当前项目差距 | 是否适合放进这一次小迭代 |
|---|---|---|
| A. 结局追踪（跨周目图鉴） | 目前有多结局，但缺“已达成清单与累计” | **是（本次采纳）**：仅改状态与引擎触发点，验收清晰 |
| B. 结局分支热力图/路径可视化 | 需要额外 UI 面板与图数据 | 否：改动面偏大 |
| C. 分支提示/后果预告系统 | 需改选项模型与文案管线 | 否：涉及设计口径较多 |

**本次只实现一项：A（结局图鉴）**。  
理由：依赖最少（仅 `PlayerPrefs` + 现有结局节点 ID）、不破坏现有剧情结构、可通过“解锁数与日志提示”直接验收。

## 6. 验收自检（Play 逻辑推演）

- 首次到达任一受追踪结局节点时，日志应出现“【结局图鉴】解锁：...（n/5）”。
- 再次到达同一结局，日志应出现“已达成”，计数不重复增长。
- 点击“再玩一次”回到 `start` 后：等级/背包按周目重置；结局图鉴计数不丢失。
- `PlayerPrefs` 键名采用 `NorthernTown2026.EndingUnlocked.<endingId>`，与现有字段无冲突。

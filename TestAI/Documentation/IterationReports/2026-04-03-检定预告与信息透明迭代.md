# 迭代报告：2026-04-03 — 检定预告与信息透明

## 1. 迭代前玩法（当时是什么样）

- **操作**：玩家在底部点击选项推进文本剧情；在右侧可拖拽物品卡到「终端/外套/饰品」槽位，或拖到「使用」区消耗（如面包）。
- **目标**：通过分支选择、属性检定与物品条件进入不同结局；并通过经验与装备提升四维属性。
- **失败/胜利**：检定采用 `属性 + 1d10 >= 阈值`，失败进入失败分支；结局节点作为阶段性“落点”，可再开新周目。
- **进度/Meta**：有等级、经验、背包、装备，以及“本局已读节点/已做选择”计数。
- **反馈现状**：检定结果会在点击后写入日志，但**点击前**选项只显示文案，不显示本次检定难度与大致成功机会，行动前信息不够透明。

## 2. 趋势要点（外部检索浓缩）

1. **行动前展示检定信息（叙事 RPG 语境）**  
   BG3 社区维基在骰子机制中明确提到：游戏会展示与检定相关的难度（DC）与成功几率信息，帮助玩家在决策前评估风险。  
   来源：<https://bg3.wiki/wiki/DC>（Dice rolls / DC / chance of success 相关段落）

2. **行动前展示战斗预测（策略游戏语境）**  
   Fire Emblem 的“Combat Forecast”长期作为核心交互，在确认攻击前展示命中等预测信息，强化“先看后做”的决策体验。  
   来源：<https://fireemblemwiki.org/wiki/Combat_forecast>

3. **通用交互原则：系统状态可见性**  
   Nielsen Norman Group 强调“Visibility of system status”：系统应及时反馈当前状态，使用户保持控制感与信任。  
   来源：<https://www.nngroup.com/articles/visibility-system-status/>

## 3. 对比结论（为什么只选这一项）

| 方向 | 与当前项目差距 | 这次是否适合做小迭代 |
|---|---|---|
| A. 选项前显示检定预告（属性/阈值/成功率） | 目前仅点击后才看到检定结果 | **是（采纳）**：只改 UI 生成逻辑，依赖少，验收明确 |
| B. 检定结果动画/音效反馈 | 当前无专门音视频反馈 | 否：需要额外资源与表现层改动 |
| C. 多资源消耗型选项（体力/金钱） | 当前 `ChoiceOption` 无消耗字段 | 否：要扩展数据结构与结算逻辑，改动面更大 |

**本次唯一实现项**：为包含 `Check` 的选项增加“检定预告”二行文案，并根据当前属性计算成功率百分比（1~10 掷骰离散概率）。

## 6. 验收自检（Play 模式或逻辑推演）

- 逻辑推演：
  - 非检定选项：仍显示单行文案，按钮高度保持原值。
  - 检定选项：显示第二行检定预告，成功率与当前属性同步变化（装备改变属性后，刷新 UI 会更新预告值）。
  - 点击后结算仍由 `TextAdventureEngine.Choose` 执行，不改变原随机检定流程，兼容旧剧情数据。
- 回归关注：
  - 仅改选项生成显示层，不改拖放、背包、周目重置逻辑。
  - 无新增持久化键，不涉及 `PlayerPrefs` 键冲突风险。

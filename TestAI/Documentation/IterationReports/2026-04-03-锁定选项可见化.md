# 迭代报告：2026-04-03 — NorthernTown2026 锁定选项可见化

## 1. 迭代前玩法（当时是什么样）

- NorthernTown2026 当前是运行时生成 UI 的文字冒险：左侧叙事日志，底部选项按钮，右侧状态/装备/背包与拖拽使用区。
- 选项推进由 `TextAdventureEngine` 驱动，支持：
  - 直接跳转；
  - 属性检定（属性 + 1d10）；
  - 条件门槛（`RequiresItemId`、`RequiresInsightSum`）；
  - 奖励经验与物品；
  - 结局回到 `start` 时新周目重置。
- **迭代前问题**：不满足条件的选项会被直接过滤，不显示给玩家。玩家只能看到“当前可点”的分支，不知道还存在什么路线、缺什么条件，复盘与重玩指向性偏弱。

## 2. 趋势要点（外部检索的浓缩）

1. **互动小说中的“Gate/Check/Set”实践**（Emily Short 博文语境）  
   在带属性/条件的分支设计里，常见做法之一是把被门槛拦住的选项“显示为灰态”而非完全隐藏，以便让玩家理解“过去行为在当前产生了后果”。  
   参考：<https://emshort.blog/2016/02/15/set-check-or-gate-a-problem-in-personality-stats/>

2. **“Failing Forward”强调减少死路、维持推进**（PF2e GM Core 规则语境）  
   即便失败也应尽量让玩家获得信息或看到下一步路径，而不是无信息中断；这和“把被锁路径显式提示”在体验目标上是一致的（减少盲试）。  
   参考：<https://2e.aonprd.com/Rules.aspx?ID=2485>

3. **平台标签语境中的“Choices Matter”**（Steam 标签页）  
   “Choices Matter”本身已成为玩家认知标签，意味着“可见的选择后果与路径差异”；把锁定路径显式化有助于贴合该语境下的可感知决策反馈。  
   参考：<https://store.steampowered.com/tags/en/Choices+matter>

## 3. 对比结论（为什么选这一项）

| 方向 | 与当前项目差距 | 这次是否适合小迭代 |
|---|---|---|
| A. 锁定选项可见化（灰显 + 条件提示） | 当前引擎已具备条件判断，只是 UI 层做了“直接隐藏” | **选中**：改动小、依赖少、验收标准清晰 |
| B. 检定失败分支全面 fail-forward 重写 | 需批量改剧情节点内容，涉及大量文本与分支设计 | 否（改动面偏大） |
| C. 新增局外 Meta（结局图鉴/持久化） | 需新增存档键、UI 区块、数据迁移考虑 | 否（本轮超出小迭代范围） |

本次只实现 A：让条件不满足的选项仍可见，但不可点击，并给出锁定原因。

## 4. 本次改动列表（改了哪些文件/行为）

1. `Assets/NorthernTown2026/Scripts/StoryModels.cs`
   - 新增 `ChoicePresentation` 模型（`Option`、`IsAvailable`、`LockedReason`），用于“选项展示层”描述。

2. `Assets/NorthernTown2026/Scripts/TextAdventureEngine.cs`
   - `GetChoicesForCurrentNode()` 改为返回 `IReadOnlyList<ChoicePresentation>`，不再只返回可点击选项。
   - 新增 `BuildChoicePresentation()`：集中生成可用性与锁定原因。
   - 对 `RequiresItemId` 不满足时生成原因：`需要物品：xxx`。
   - 对 `RequiresInsightSum` 不满足时生成原因：`需要洞察+机巧 ≥ N（当前 M）`。
   - `Choose()` 增加二次防护：若外部绕过 UI 传入不可用选项，记录日志并拒绝执行，避免状态错乱。

3. `Assets/NorthernTown2026/Scripts/TextAdventureBootstrap.cs`
   - 选项重建逻辑改为遍历 `ChoicePresentation`：
     - 可用选项：原有样式、可点击；
     - 锁定选项：灰态底色/文字、不可点击；
     - 文案追加 `（未满足：...）` 条件提示。
   - 增加空值保护：仅在 `view.IsAvailable && opt != null` 时绑定点击事件，避免无效引用。

4. `Assets/NorthernTown2026/GAMEPLAY_FRAMEWORK.md`
   - 将“选项可见性”更新为与实现一致：**全部显示，不满足条件则置灰并显示原因**，避免文档与代码偏差。
